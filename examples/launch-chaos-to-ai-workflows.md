# Example: Launch-stage founder overwhelmed by feedback

## User prompt

```text
Use AI-Native Founder Orchestrator.

We launched last month.
Product: AI meeting follow-up tool for consultants.
Inputs arriving:
- 40 support emails per week
- bug reports in Slack
- feature requests in calls
- 12 sales calls with notes in Google Docs
- founder still manually reads everything

Turn my launch chaos into AI workflows.
```

## Expected output style

```text
Stage: Launch

Why this stage:
The product has real users and repeated feedback loops. The bottleneck is no longer only building; it is founder attention, prioritization, and operating rhythm.

Founder-only decisions:
- Decide which customer segment gets priority support.
- Decide which feedback changes positioning versus roadmap.
- Decide quality standards for customer-facing AI outputs.

Claude Chat tasks:
- Summarize weekly themes from support, bugs, and sales notes.
- Separate PMF signals from novelty, politeness, and edge cases.
- Draft response templates and prioritization rules.

Claude Code tasks:
- Build or script a feedback inbox with tags: bug, confusion, activation, sales objection, feature request.
- Add analytics for repeated use and follow-up completion.
- Create test coverage for the core follow-up generation workflow.

Claude Cowork tasks:
- Read sales notes, support threads, and call transcripts.
- Extract repeated objections, customer language, and workflow gaps.

Skills / memory / workflow opportunities:
- Weekly launch review Skill.
- Sales call synthesis memory.
- Support triage workflow.

This week's artifact:
AI workflow map.

Exit criteria:
- Founder reviews a synthesized attention queue, not every raw message.
- Top 5 recurring issues are visible.
- Product decisions connect to repeated customer evidence.

AI-native failure mode:
Failure: founder attention becomes the hidden operating system.
Countermeasure this week: define one weekly workflow where AI turns raw inputs into a ranked decision queue.
```
